### 1.3 数据流转流程

**学员搜索流程：**
```
用户输入姓名 → 防抖 250ms → 调用 /api/students?q=姓名
  → Edge Function 读取 Blob: students/index.json
  → 精确匹配优先 + 模糊匹配补充
  → 返回学员列表 → 前端下拉展示
```

**排课查询流程：**
```
选中学员 + 当前视图日期范围
  → 调用 /api/schedules?studentId=xxx&startDate=...&endDate=...
  → Edge Function 调用 store.list() 获取该学员所有月份文件
  → 并行读取各月排课数据 → 按日期范围过滤
  → 返回排课列表 → 前端按视图渲染日历
```

**视图切换流程：**
```
切换月/周/日视图 → 重新计算日期范围 → 触发排课重新加载
  → useMemo 缓存日期计算 → 仅重新渲染视图组件
```

## 二、数据库设计

### 2.1 存储方案选型分析

| 对比项 | Edge KV | Edge Blob（选用） |
|--------|---------|-------------------|
| 数据组织 | 扁平 key→value | 按 `/` 路径组织，支持目录层级 |
| Key 限制 | 仅字母数字下划线 | 任意字符串路径 |
| 一致性 | 60s 最终一致 | 支持强一致模式 |
| 运行环境 | 仅 Edge Functions | Edge + Cloud Functions |
| 配置方式 | 控制台开通→创建→绑定 | `getStore()` 即用即得 |
| 遍历能力 | `list()` 列出 key | `list({prefix})` 按目录遍历 |

**选择 Blob 的核心原因：**
1. **目录层级天然适配**：排课数据按 `schedules/{studentId}/{yyyy-MM}.json` 组织，按学员+月份查询直接定位文件，无需遍历
2. **强一致模式**：排课修改后立即生效（KV 最长 60s 延迟）
3. **Key 无字符限制**：可直接用学员 ID（含字母数字）作为路径，无需编码
4. **零配置**：`getStore('schedule-system')` 自动创建命名空间，部署最简

### 2.2 数据模型

#### 学员表（students/index.json）

```json
[
  {
    "id": "s001",
    "name": "张伟",
    "phone": "13800001001",
    "grade": "高三"
  }
]
```

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 学员唯一标识，主键 |
| name | string | 学员姓名，用于精确/模糊搜索 |
| phone | string | 联系电话（可选） |
| grade | string | 年级（可选） |

#### 排课表（schedules/{studentId}/{yyyy-MM}.json）

按学员 ID 分目录、按月份分文件存储，每个文件为该学员该月的排课数组。

```json
[
  {
    "id": "sch_0001",
    "studentId": "s001",
    "studentName": "张伟",
    "courseName": "数学提高班",
    "teacher": "张老师",
    "location": "A教室201",
    "date": "2026-07-15",
    "startTime": "08:00",
    "endTime": "09:30",
    "note": ""
  }
]
```

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 排课记录唯一标识 |
| studentId | string | 关联学员 ID（外键） |
| studentName | string | 学员姓名（冗余存储，避免跨文件 join） |
| courseName | string | 课程名称 |
| teacher | string | 授课教师 |
| location | string | 上课地点 |
| date | string | 上课日期，格式 yyyy-MM-dd |
| startTime | string | 开始时间，格式 HH:mm |
| endTime | string | 结束时间，格式 HH:mm |
| note | string | 备注（可选） |

### 2.3 存储结构图

```
Blob Store: schedule-system/
├── students/
│   └── index.json                    # 全部学员列表
└── schedules/
    ├── s001/
    │   ├── 2026-07.json              # 张伟 7月排课
    │   ├── 2026-08.json              # 张伟 8月排课
    │   └── ...
    ├── s002/
    │   ├── 2026-07.json              # 李娜 7月排课
    │   └── ...
    └── ...
```

### 2.4 设计要点

- **按学员分目录**：按学员查询时直接 `list({ prefix: 'schedules/s001/' })`，无需扫描全表
- **按月份分文件**：日期范围查询时仅需读取对应月份数据，减少数据传输量
- **冗余 studentName**：排课记录中冗余存储学员姓名，避免展示时额外读取学员表
- **JSON 格式**：结构化数据可读性好，`JSON.parse` 解析快，单文件上限 25MB 足够

核心函数 getMonthCells(date, schedules):

## 五、性能优化策略

### 5.1 数据加载优化

| 策略 | 实现方式 | 效果 |
|------|----------|------|
| **按月分文件** | Blob 路径 `schedules/{id}/{yyyy-MM}.json` | 日期范围查询仅读取对应月份，减少数据传输 |
| **并行读取** | `Promise.all(months.map(readMonth))` | 多月数据并行加载，总延迟≈单次延迟 |
| **防抖搜索** | 输入后 250ms 延迟请求 | 减少无效 API 调用 70%+ |
| **请求超时** | `AbortSignal.timeout(3000)` | 3 秒超时快速回退 mock，避免白屏 |
| **边缘缓存** | EdgeOne CDN 缓存静态资源 | 全球节点就近响应，延迟 <50ms |
| **强一致按需** | Blob 默认最终一致，排课写入用 strong | 平衡性能与一致性 |

### 5.2 视图渲染优化

| 策略 | 实现方式 | 效果 |
|------|----------|------|
| **useMemo 缓存** | 缓存日期计算、统计数据 | 避免重复计算 |
| **useCallback** | 缓存事件处理函数 | 避免子组件不必要重渲染 |
| **月视图截断** | 每格最多渲染 3 条排课 | DOM 节点数量可控 |
| **虚拟列表预备** | 排课超 100 条可引入虚拟滚动 | 大数据量场景可扩展 |
| **CSS 过渡** | `transition-all` + `hover:scale` | 流畅交互无卡顿 |
| **Tailwind 纯净化** | `content` 配置精确扫描 | 移除未使用 CSS，产物更小 |

### 5.3 构建优化

- Vite 构建自动代码分割（动态 import）
- 静态资源哈希命名，配合 CDN 长缓存
- Tree-shaking 移除未使用代码
- date-fns 按需引入，仅打包用到的函数

## 六、安全考虑

### 6.1 用户认证（当前版本）

当前为查询展示型系统，暂无写入操作（种子数据通过 POST /api/seed 初始化）。
后续如需增加排课管理功能，建议方案：

| 方案 | 实现方式 | 适用场景 |
|------|----------|----------|
| **EdgeOne Access** | 平台内置访问控制 | 内部系统、仅授权用户可见 |
| **Token 认证** | 登录后签发 JWT，存入 KV/Session | 需要用户体系 |
| **Basic Auth** | 边缘函数校验 Authorization 头 | 简单场景 |

### 6.2 数据权限控制

- **读权限**：当前版本为只读展示，数据无敏感信息
- **写权限**：`/api/seed` 仅用于初始化，生产环境应移除或加鉴权
- **CORS**：API 响应头设置 `Access-Control-Allow-Origin`，可按需收紧为指定域名

### 6.3 平台级安全

EdgeOne Makers 免费版自带：
- **DDoS 防护**：自动化流量清洗
- **速率限制**：自定义速率限制规则
- **IP 黑白名单**：控制访问来源
- **Referer/UA 黑白名单**：防盗链与爬虫防护
- **HTTPS 强制**：全站 HTTPS，SSL 证书自动签发

### 6.4 数据安全

- Blob 存储仅在 Edge Functions 内部访问，不暴露公网
- 学员手机号等敏感信息可选不展示
- 建议生产环境对手机号脱敏显示（如 `138****1001`）

## 九、关键功能模块实现思路与技术选型说明

### 9.1 日历视图 — 为什么不用第三方日历库？

本系统**自主实现**日历视图，而非使用 fullcalendar 等第三方库，原因：

| 对比项 | 第三方库（fullcalendar） | 自主实现（选用） |
|--------|-------------------------|------------------|
| 包体积 | 200KB+ | 0 额外依赖 |
| 定制性 | 受 API 限制 | 完全可控 |
| 样式一致性 | 需覆盖默认样式 | TailwindCSS 原生 |
| EdgeOne 适配 | 可能有兼容问题 | 纯 React 组件，零问题 |
| 维护性 | 升级风险 | 代码可控 |

**核心实现依赖 date-fns** 提供日期计算（`startOfMonth`/`endOfWeek`/`eachDayOfInterval` 等），视图层完全自主控制。

### 9.2 学员查询 — 精确+模糊双模式

后端在单个 API 中同时实现两种匹配：

```javascript
// 精确匹配优先
const exact = students.filter(s => s.name === q)
// 模糊匹配补充（排除已精确匹配的）
const fuzzy = students.filter(s => s.name !== q && s.name.includes(q))
return [...exact, ...fuzzy]
```

**优势**：一次请求完成两种搜索，精确结果始终排在前面，用户体验更好。

### 9.3 存储 — 为什么选 Blob 而非 KV？

详见「二、数据库设计」的选型分析表。核心优势：目录层级 + 强一致 + 零配置。

### 9.4 前端 — 为什么用 mock 回退机制？

Edge Functions 本地开发需要 EdgeOne CLI，对新手有门槛。mock 回退机制让前端 `npm run dev` 即可独立预览，部署后自动切换到真实 API，**开发体验与部署体验双赢**。

## 十、后台管理模块

后台管理提供数据全生命周期管理能力，入口位于日历视图右上角「后台管理」按钮。

### 10.1 功能概览

| 功能 | 说明 | 对应 API |
|------|------|----------|
| 初始化种子数据 | 一键写入 8 名示例学员及 7 月排课，用于演示验证 | `POST /api/seed` |
| 一键清空数据 | 删除 Blob 中全部学员与排课，二次确认防误操作 | `POST /api/clear` |
| JSON 数据导入 | 文本框粘贴或上传 .json 文件，支持 merge/replace 模式 | `POST /api/import` |
| 排课列表管理 | 按学员查看排课列表，支持编辑/删除单条 | `GET /api/schedules` |
| 排课编辑 | 表单编辑排课，自动处理跨月/跨学员存储迁移 | `PUT /api/schedule-update` |
| 排课删除 | 删除单条排课，空文件自动清理 | `DELETE /api/schedule-delete` |

### 10.2 排课修改的跨月存储路径处理

排课按 `schedules/{studentId}/{yyyy-MM}.json` 分文件存储。修改日期或学员时，存储路径可能变化，处理逻辑如下：

| 情况 | 触发条件 | 处理方式 |
|------|----------|----------|
| 原地更新 | 学员与日期月份均未变 | 读出当月数组 → 替换对应记录 → 写回原文件 |
| 跨月迁移 | 日期月份变化（如 7月→8月） | 从旧月文件删除 + 写入新月文件 + 旧文件空则删除 |
| 跨学员迁移 | 学员变更 | 从旧学员目录删除 + 写入新学员目录对应月份 |
| 跨学员+跨月 | 两者都变 | 同上，路径同时变化 |

**实现位置：** [functions/_lib/store.js](file:///c:/Users/czg95/Documents/code/pai/functions/_lib/store.js) - `updateSchedule()`

**前端提示：** 编辑表单检测到跨月/跨学员时，会显示 amber 色提示框，展示旧路径→新路径的迁移预览，用户确认后才提交。

### 10.3 JSON 导入模式

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| `merge`（追加合并） | 保留已有数据，按 id 去重，新数据覆盖同 id 旧数据 | 增量导入、小批量调整 |
| `replace`（替换） | 清空后写入新数据 | 全量数据更新、重新初始化 |

### 10.4 安全考虑

- 清空操作需**二次确认**（浏览器 confirm 对话框）
- 删除单条排课需**一次确认**
- 管理类 API 不回退 mock（避免误操作本地数据）
- 建议生产环境为 `/api/seed`、`/api/clear`、`/api/import`、`/api/schedule-*` 增加鉴权（见第九章安全考虑）
